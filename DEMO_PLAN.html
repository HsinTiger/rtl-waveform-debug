🔴 RTL behavior 電路設計

25–35%

Corner case、介面協議不一致、operating mode 路徑。嚴重性最高。跨 config 都 fail → A 高度可疑。

B

🟠 TXCMP/RXCMP Checker

30–45%

最大宗。比對時間窗錯、off-by-one、latency mismatch。紅燈 > 50% 是 checker 在叫，不是電路在錯。

C

🔵 sim_setting.json

10–20%

DV 拿到舊版/錯誤 config，或參數組合非預期。最快確認（diff 2 分鐘），通常 DV 自己就先發現了。

D

🟣 phyUD C++ 實作

5–15%

多數 fixed-point 精度（truncation vs rounding）。少數才是 behavior 寫錯。跨多個 checker 同時 fail → D。

🔍 Check Order（最短 debug 路徑）

Step 1: Config Diff → 有差異 → 結案 ▸ 重新拿 json → 一致 → 繼續
Step 2: Checker Timing → 有誤 → 結案 ▸ 修正 checker → 無誤 → 繼續
Step 3: RTL vs Spec 比對 → RTL ≠ Spec → A（RTL bug） → RTL = Spec → D（C++ bug）

📋 Agent 最終輸出格式（三大方向 + 概率）

[高概率 55%] TX/RX Data Mismatch → tx_valid 邏輯 shift 1 cycle → 看 RTL line 145-180 2. [中概率 25%] Checker Timing（TXCMP） → exp/act 差 1 cycle → 看 checker 的 latency parameter 3. [低概率 20%] sim_setting.json → freq_ratio 可能不對 → diff 確認 _概率總和 90%・保留 10% 給未預期錯誤_
